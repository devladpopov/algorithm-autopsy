# Algorithm Audit: формализация задачи

## 1. Цель

Систематический поиск математических ошибок и багов реализации в классических алгоритмах (опубликованных 30+ лет назад), которые широко используются, но не проходили формальную верификацию.

Прецедент: Flouri et al. (2015) обнаружили две ошибки в алгоритме Gotoh (1982). Проверка 10 реализаций показала, что только 2 дают корректный результат. 8 из 31 учебных материалов содержали ошибку.

## 5. Методология аудита

### Этап 1: Верификация по оригинальной статье
1. Найти оригинальную публикацию (не учебник)
2. Реализовать алгоритм строго по формулам из статьи
3. Сравнить с 5-10 существующими реализациями на случайных входах
4. Расхождения = признак бага минимум в одной реализации

### Этап 2: Property-based тестирование
Для каждого алгоритма определить математические инварианты:
- **Выравнивание последовательностей**: score(A,B) == score(B,A)? Оптимальность? Пустые строки? Одинаковые строки?
- **Графовые**: triangle inequality? Корректность при negative weights? Consistency с brute force на малых графах?
- **Сортировки**: стабильность? Корректность на дубликатах? Пустые/одноэлементные массивы?
- **Строковые**: все вхождения найдены? Overlapping matches? Пустой паттерн?

### Этап 3: Граничные условия и инициализация
Главный источник ошибок (обе ошибки Gotoh были здесь):
- Проверить инициализацию матриц/структур данных
- Проверить индексацию (0-based vs 1-based)
- Проверить граничные случаи: пустой вход, один элемент, максимальный размер

### Этап 4: Числовая стабильность
- Очень длинные последовательности / большие графы
- Экстремальные значения параметров (штрафы, веса)
- Переполнения (integer overflow, float infinity)
- Потеря точности при float-арифметике
- Деление на числа близкие к нулю

### Этап 5: Формализация результатов
Для каждого найденного бага:
- Описание ошибки
- Минимальный воспроизводимый пример
- Какие реализации затронуты
- Корректное решение
- Pull request / issue в затронутые репозитории
- Публикация (bioRxiv/arXiv preprint)

## 8. Следующие шаги

1. Выбрать 1-2 алгоритма из Tier 1 для первого аудита
2. Собрать оригинальные статьи + 5-10 реализаций для каждого
3. Написать property-based тесты
4. Провести аудит по методологии (этапы 1-5)
5. Оформить результаты

Рекомендация: начать с **BLAST** (A4) или **Boyer-Moore** (E2), так как у них максимальное число реализаций для сравнения и широчайшее практическое применение.
